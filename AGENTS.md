# Essential Whenever Instructions

## Codebase overview

This is a Python datetime library with a pure-Python implementation (`pysrc/whenever/_pywhenever.py`)
and a Rust extension (`src/`) that mirrors it for performance. The Python implementation is the
**reference**: match its behavior exactly when porting to Rust.

Key directories:
- `pysrc/whenever/_pywhenever.py` — canonical Python implementation (single large file)
- `src/classes/` — one file per Python type (date.rs, time_delta.rs, zoned_datetime.rs, etc.)
- `src/common/` — shared Rust logic (scalar types, formatting, parsing, rounding, math)
- `src/py/` — safe wrappers around `pyo3_ffi`
- `src/pymodule/def.rs` — module definition, `State` struct, type registration
- `src/docstrings.rs` — **autogenerated**, do not edit directly. Change docstrings in `_pywhenever.py` and run `scripts/generate_docstrings.py`
- `tests/` — Python tests (comprehensive); Rust tests are minimal (only for core logic)

## Commands

The `Makefile` is the authoritative interface for project workflows. Use its targets rather
than invoking their underlying tools directly, unless diagnosing a target.

For compact agent output, use `make QUIET=1 <target>`. This suppresses echoed recipes and
uses concise pytest and Cargo output. Customize test runs without replacing their workflow:

- `TEST_PATH` selects the tests to run (for example, `TEST_PATH=tests/test_date.py`)
- `PYTEST_ARGS` overrides pytest flags
- `CARGO_ARGS` overrides Cargo flags

## General reminders

- Don't forget to update the changelog (for new/breaking features)
- When changing the public API, also update the type stubs (.pyi file)
- Code coverage for Python code must be complete, with exceptions only for
  hard-to-test cases involving (for example) system configuration.
- Don't put things in docstrings that are already described in the type annotations.

## Development philosophy

- **Remove redundant checks**: if a condition is guaranteed by earlier logic, don't re-check it.
  Add a debug assert and/or a comment explaining why it's safe instead.
- **Error message capitalization**: use lowercase or error messages.
- Only comment code that needs clarification. Do not comment otherwise.

## Tests

- Run with `make test-py` (not bare `pytest` — avoids benchmark tests)
- Run Rust-only unit tests with `make test-rs`
- **Pure Python tests**: remove the compiled extension first (`make clean-ext`)
  Rebuild after with `make build`.
- Group related test functions in a class when testing a single method/concept
- Make liberal use of `pytest.mark.parametrize` to describe repeated test cases without duplication
- Match existing test style and keep tests in the same file when possible
- Pre-existing test failures are common (features not yet ported to Rust). Ignore them
  unless they're related to your changes.
